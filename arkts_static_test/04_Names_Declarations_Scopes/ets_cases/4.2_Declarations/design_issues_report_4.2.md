## 一、行为差异与规范一致性概览

### 问题 A-4.2：预定义类型冲突检测不完整 ⭐⭐ MEDIUM

**用例：** NAM_04_02_103_FAIL_predefined_type_clash

**实际行为：** 编译器仅检测与 `number`、`String`、`Record`、`Object`、`Array` 的同名冲突，但未检测与 `int`、`Box` 等编译器内部保留名的冲突（直到编译时才报错）。

**Spec 依据：** §4.2 未明确列举所有预定义类型，导致保留名管理不一致。

**对比：**
| 语言 | 预定义类型冲突检测 |
|------|-------------------|
| ArkTS | 部分检测（Spec 列举的），运行时才检测编译器保留名 |
| Java | 编译期检测与 java.lang 类型同名 |
| Swift | 标准库类型可遮蔽（需显式模块名） |
| TypeScript | 允许遮蔽（不推荐） |

**影响：** 用户需要多次编译才能发现所有命名冲突。

**建议：** 统一保留名管理，要么全部在 Spec 中列举，要么全部由编译器动态检测。

---

### 差异 B-4.2：泛型擦除后重载不可区分（语言设计差异）

**Observation:** ArkTS 在类型擦除后检测重载是否可区分，如果擦除后签名相同则报错。Java 允许类型擦除后不可区分的重载（编译通过但不可共存）。

**对比：**
| 语言 | 泛型擦除重载 |
|------|-------------|
| ArkTS | ❌ 类型擦除后不可区分时报错 |
| Java | ⚠️ 编译通过但不可共存（运行时桥接方法） |
| Swift | ✅ 泛型特化可区分 |
| TypeScript | N/A（无泛型擦除） |

**影响：** ArkTS 的重载规则更严格，防止运行时歧义。

**评价：** 符合 ArkTS 静态化设计哲学，非缺陷。

## 二、跨语言对比详细分析

### 2.1 同名声明规则

| 维度 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 同一作用域同名 | ❌ 禁止（overload 除外） | ❌ 禁止（overload 除外） | ❌ 禁止（重载需参数标签区分） | ✅ 允许（后者覆盖前者） |
| 不同作用域同名 | ✅ 遮蔽（shadowing） | ✅ 遮蔽 | ✅ 遮蔽 | ✅ 遮蔽 |
| 重载可区分性 | 基于参数类型序列 | 基于参数类型/数量 | 基于参数类型/标签 | N/A（不支持重载） |

### 2.2 预定义类型冲突

| 维度 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 检测时机 | 编译期（部分） | 编译期 | 编译期 | 运行时 |
| 冲突处理 | 编译错误 | 编译错误 | 需显式模块名 | 允许遮蔽 |
