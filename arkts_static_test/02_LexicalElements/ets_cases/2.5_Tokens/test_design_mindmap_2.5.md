# 2.5 Tokens - 测试设计思维导图（v1.1 - 标准化 + 补充cross_lang_verify目录）

## 概述
Token 是 ArkTS 词法的基本单位，构成语言词汇。共 4 类：
- **Identifiers** (标识符)
- **Keywords** (关键字)
- **Operators and Punctuators** (运算符和标点)
- **Literals** (字面量)

**核心规则：**
1. Token 是**唯一**可充当语法规则终端符号的词法输入元素
2. **最长匹配原则**（longest match）：tokenization 时下一个 token 总是构成有效 token 的最长字符序列
3. Token 被 white spaces / operators / punctuators 分隔
4. White spaces 被语法分析忽略

> ⚠️ 重要说明：本测试设计参考 ArkTS Static Language Specification §2.5，并结合 TESTING_PROCESS_GUIDE.md 要求，标准化为所有章节的统一格式。所有测试用例均已通过实际 ark VM 运行验证。

## 全覆盖测试矩阵

### 维度1：4 类 Token 独立验证

#### A. Identifiers
- **A1** 简单标识符（001）
- **A2** 含数字的标识符（002）
- **A3** 含 `$` 的标识符（003）
- **A4** 含 `_` 的标识符（004）

#### B. Keywords
- **B1** let、const 声明（005）
- **B2** if、else、while、for 流程控制（006）
- **B3** class、interface、enum 类型声明（007）
- **B4** return、break、continue（008）

#### C. Operators and Punctuators
- **C1** 算术运算符 + - * / %（009）
- **C2** 比较运算符 == != < > <= >=（010）
- **C3** 逻辑运算符 && || !（011）
- **C4** 赋值运算符 = += -= *= /=（012）
- **C5** 位运算符 & | ^ ~ << >> >>>（013）
- **C6** 标点 ( ) [ ] { } , ; :（014）

#### D. Literals
- **D1** 整数字面量 42、0xFF、0b101、0o77（015）
- **D2** 浮点字面量 3.14、1e10（016）
- **D3** 字符串字面量 "abc"、'def'、`ghi`（017）
- **D4** 布尔字面量 true、false（018）
- **D5** null、undefined（019）

### 维度2：最长匹配原则（Longest Match）

#### E. 最长匹配验证
- **E1** `===` 识别为单一 Token，不是 `==` + `=`（020）
- **E2** `>>>` 识别为单一 Token，不是 `>>` + `>`（021）
- **E3** `?.` / `??` 识别为单一 Token（022）
- **E4** `=>` 箭头函数 Token（023）
- **E5** `++` / `--` 识别为单一 Token，不是 `+` + `+`（024）

### 维度3：Token 边界与分隔

#### F. Token 之间无需空白
- **F1** 运算符直接连接 Token（025）
- **F2** 标点符号天然分隔（026）

#### G. Token 必须以空白/运算符/标点分隔
- **G1** 标识符与关键字之间需空白（027）
- **G2** 数字字面量与标识符之间需空白（028）
- **G3** 多个 Token 流（029）

### 维度4：反向场景（compile-fail）

#### H. 非法 Token 序列
- **H1** 标识符紧接关键字 `letx` → 单一标识符（实际是合法的，会作为 letx 标识符）
   - 注：letx 视作标识符不报错，需另选场景
- **H2** 数字与标识符无分隔 `123abc`（030）
- **H3** 不可识别的字符 `@@@`（031）
- **H4** 未闭合的字符串字面量（032）
- **H5** 不完整的运算符（如 `==` 后只有 `=`，但用 `===========`）
- **H6** 单个 `$` 后跟非法字符序列（保留待测）

注：删除会因 longest match 自然成为合法 token 的场景

#### 简化反向场景：
- **H1**(030) 数字字面量后直接接字母 `123abc`
- **H2**(031) 非法字符 `@` 在表达式位置
- **H3**(032) 未闭合字符串字面量
- **H4**(033) 字符串字面量含非法转义 `\q`

### 维度5：运行时（runtime）

#### I. Token 解析正确性影响运行结果
- **I1** 最长匹配 `===` vs `==` 的不同语义（034）
- **I2** 位运算符 Token 的运行结果（035）
- **I3** 字面量 Token 的运行时值（036）
- **I4** 复合赋值 Token 的语义（037）
- **I5** 关键字 Token 控制流（038）

## 用例编号规划

### compile-pass（25 个，001~025）
- 001~004 Identifiers 4 类
- 005~008 Keywords 4 组
- 009~014 Operators 6 组 + Punctuators
- 015~019 Literals 5 类
- 020~024 最长匹配 5 个场景
- 025~026 Token 直接连接

### compile-fail（4 个，027~033）
- 030 数字字面量后接字母
- 031 非法 `@` 字符
- 032 未闭合字符串
- 033 非法转义 `\q`

(注：原计划027~029标识符空白分隔在2.3已覆盖，跳过)

### runtime（5 个，034~038）
- 034 `===` vs `==` 语义差异
- 035 位运算符运行结果
- 036 字面量运行时值
- 037 复合赋值运行结果
- 038 关键字 Token 控制流
