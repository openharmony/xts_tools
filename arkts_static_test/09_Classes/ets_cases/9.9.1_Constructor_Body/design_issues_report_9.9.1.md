## 三、值得关注的设计观察

### 观察 A：ArkTS 构造器体执行顺序设计严密 ⭐ DESIGN

**spec §9.9.1 详细规定了主构造器体的五步执行顺序：**

1. 不涉及 `this`/`super` 的可选代码
2. 调用父类构造器 `super()`
3. 编译器隐式添加字段初始化器（按声明顺序）
4. 不通过 `this` 访问未初始化字段的代码
5. 所有字段初始化完成后的任意代码

**用例验证：** CLS_09_09_011 通过三级继承链精确追踪执行顺序，输出 `"GfGcPfPcCfCc"`（父类字段→父类ctor→子类字段→子类ctor→孙类字段→孙类ctor），与 spec 示例输出完全一致。

**对比：**

| 语言 | 构造器执行顺序规范化 |
|------|-------------------|
| ArkTS | ✅ spec 明确定义五步顺序 |
| Java | ✅ JLS 明确但语义有差异（字段初始化位置不同） |
| TypeScript | ❌ 无此级别规范，依赖 ES 运行时 |

**评价：** ArkTS 在构造器体执行顺序上进行了显式规范化，比 TypeScript 更严格、更可预测。

### 观察 B：super() 前代码限制兼顾灵活性与安全性 ⭐ DESIGN

**spec 明确允许** super() 调用前放置 "arbitrary code that uses neither this nor super"。

**用例：** CLS_09_09_010 在 super() 前进行局部变量计算和静态代码，编译通过。

**对比：**

| 语言 | super() 前代码限制 |
|------|------------------|
| ArkTS | ✅ 允许不涉及 this/super 的代码 |
| Java | ✅ 允许（但更宽松，可访问 static 成员） |
| Swift | ❌ 两阶段初始化，phase 1 极其严格 |

**评价：** ArkTS 在安全（禁止访问未初始化的 this）和灵活（允许预处理计算）之间取得了良好平衡。

### 观察 C：编译时检测非初始化字段访问的多级策略 ⭐ DESIGN

**spec 采用三级策略处理字段初始化检测：**

1. 编译器可证实时 → compile-time error
2. 可能违反但不确定时 → compile-time warning
3. 无法静态判定时 → 运行时行为由实现决定

**评价：** 这是一种务实的工程折中，在不牺牲太多表达能力的前提下最大化静态安全。
