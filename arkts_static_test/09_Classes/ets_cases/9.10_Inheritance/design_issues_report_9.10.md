## 三、值得关注的设计观察

### 观察 A：ArkTS 要求 override 关键字显式声明 ⭐ DESIGN

**spec §9.10 明确：** 派生类重写基类方法/字段时，必须使用 `override` 关键字显式声明。

**对比：**

| 语言 | override 要求 |
|------|--------------|
| ArkTS | ✅ 必须显式 override |
| Java | ✅ @Override 注解（推荐非强制） |
| TypeScript | ✅ override 关键字（TS 4.3+，需开启 noImplicitOverride） |
| Swift | ✅ override 关键字（必须显式） |
| C# | ✅ override 关键字（必须显式） |

**评价：** ArkTS 与 Swift/C# 一致，强制显式 override 可防止意外重写，属于良好设计实践。

### 观察 B：字段 override 要求类型严格相同 ⭐ DESIGN

**用例：** CLS_09_10_016_PASS_FIELD_OVERRIDE_SAME_TYPE / CLS_09_10_017_FAIL_FIELD_OVERRIDE_TYPE_MISMATCH

派生类同名字段类型必须与基类字段类型完全相同，不满足则编译失败。

**对比：**

| 语言 | 字段覆盖规则 |
|------|------------|
| ArkTS | 类型必须相同 |
| Java | 字段隐藏（field hiding），不要求类型相同 |
| TypeScript | 无编译错误，类型可不同 |
| Swift | 属性重写类型必须兼容 |

**评价：** ArkTS 对字段 override 的类型要求比 Java 更严格（Java 使用 shadowing），这避免了运行时类型混淆，属于安全性增强设计。

### 观察 C：override 访问修饰符必须一致 ⭐ DESIGN

**用例：** CLS_09_10_019_FAIL_FIELD_OVERRIDE_ACCESS_MODIFIER_MISMATCH

重写字段的访问修饰符必须与基类被重写字段完全相同。

**评价：** 这比 Java（可放宽访问权限）和 C#（可放宽但不能收窄）更严格。ArkTS 选择了最保守的策略——修饰符完全不可改变，避免了子类改变成员可见性带来的语义模糊。

### 观察 D：static 方法不通过实例继承 ⭐ DESIGN

**用例：** CLS_09_10_035_FAIL_STATIC_NOT_INHERITED / CLS_09_10_024_RUNTIME_STATIC_NOT_INHERITED

父类 static 方法只能通过父类名显式限定调用，不能通过子类实例访问。

**对比：**

| 语言 | static 继承行为 |
|------|----------------|
| ArkTS | 不通过实例继承，仅通过类名访问 |
| Java | 可通过子类名/实例引用（有警告） |
| TypeScript | 可通过子类名访问 |
| Swift | 可通过子类名访问（继承） |

**评价：** ArkTS 的严格限制消除了 Java 中常见的 static 方法"伪继承"混淆，属更清晰的设计。
