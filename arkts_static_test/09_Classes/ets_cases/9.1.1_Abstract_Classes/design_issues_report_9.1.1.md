# 9.1.1 Abstract Classes - ArkTS与Java/Swift/TS行为差异及规范一致性报告

**报告日期：** 2026-06-22
**测试用例数：** 12（compile-pass: 4, compile-fail: 5, runtime: 3）
**目的：** 通过用例执行（编译期 + 真实运行时）+ spec 对比，识别 ArkTS 抽象类的设计问题

---

## 一、与业界静态语言的差异点

**无严重设计问题发现。** 所有 12 个用例的行为均与 ArkTS spec 保持一致。但有一个设计观察点：

### 设计观察：abstract + override 组合禁止

**用例：** CLS_09_01_020_FAIL_ABSTRACT_METHOD_OVERRIDE
**对比：** ArkTS 禁止 `abstract override` 组合，而 Java 允许抽象子类重新声明抽象方法（`abstract override` 表示"将父类方法重新声明为抽象"）。Swift 的 protocol 方法可以在子类中重新声明。

**影响：** 如果需要中间抽象层（如 B extends A, B 将 A 的方法重新声明为 abstract），当前 ArkTS 不支持这种模式。但这是 spec 明确的设计选择，非缺陷。

## 五、跨语言对比

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| **抽象类声明** | ✅ | ✅ | ❌（用protocol） |
| **实例化禁止** | ✅ | ✅ | ✅ |
| **abstract+final** | ❌ 拒绝 | ❌ 拒绝 | ❌（无abstract） |
| **abstract+override** | ❌ 拒绝 | ✅ 允许 | ✅ 允许（protocol重写） |
| **多态派发** | ✅ | ✅ | ✅ |

**结论：** 9.1.1 抽象类规则与 Java 高度一致，唯一差异是 abstract+override 组合。Swift 用 protocol 替代 abstract class，功能等价但语法不同。

## 七、改善建议

### 短期
1. 无需修改 — 当前实现与 spec 完全一致。

### 中期
2. 考虑增加更多抽象类边界测试（如 abstract+static 组合、多层抽象继承链中的构造器执行顺序）。

### 长期
3. 评估是否放开 abstract+override 组合 — Java 允许此组合用于中间抽象层设计模式。
