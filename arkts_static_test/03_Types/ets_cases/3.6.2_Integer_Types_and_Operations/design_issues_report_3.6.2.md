### 问题 U：**模块级 const 整数除零未做编译期检测**（实现 bug）⭐⭐⭐ HIGH ⭐ NEW

**用例：** TYP_03_06_02_028_RUNTIME_INT_DIV_MODULE_CONST_ZERO（实测发现）
**类别：** C 类（编译器实现 bug）+ D 类（spec 未明确）

**实测对比：**

| 场景 | 用例 | 预期 | 实测 | 一致性 |
|------|------|------|------|--------|
| `let c = 10 / 0`（字面量） | 022 | 编译错误 | ✅ ESY0273 | ✅ |
| 函数内 `const a = 0; 10/a` | 024 | 编译错误 | ✅ ESY0273 | ✅ |
| 模块级 `const a = 0; 10/a` | 028 | 编译错误（一致性预期）| ❌ **编译通过** | ❌ **不一致 bug** |
| `10 / (1-1)`（常量表达式）| 025 | 编译错误 | ✅ ESY0273 | ✅ |

**详细描述：**
```typescript
// 场景 A：函数内 const → 编译错误
function fnA(): void {
  const a: int = 0
  let c: int = 10 / a   // ❌ ESY0273: Division by zero is not allowed
}

// 场景 B：模块级 const → 编译通过，运行时抛
const moduleA: int = 0
function fnB(): void {
  let c: int = 10 / moduleA   // ⚠️ 编译通过，运行时抛 ArithmeticError
}
```

**🔴 跨语言对比：**

| 行为 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 函数内 `const a = 0; 10/a` | ❌ 编译错误 | ✅ 通过（运行时抛）| ❌ 编译错误 |
| 模块级 `const a = 0; 10/a` | ✅ **通过（bug）** | ✅ 通过（运行时抛）| ❌ 编译错误 |
| 一致性 | ❌ **不一致** | ✅ 一致 | ✅ 一致 |

**结论：**
- Java 设计：完全不做编译期检测（一致）
- Swift 设计：所有 const/let 都做编译期检测（一致）
- ArkTS：**部分做检测**（仅函数内 const），模块级 const 漏检（**不一致**）

**影响：**
1. 同样的 `const a: int = 0`，仅因声明位置不同行为完全不同
2. 用户难以预测哪些情况会编译错误
3. spec §15.17.2 未明确"detected during compilation"的范围

**建议：**
1. **统一行为**：要么模块级 const 也做检测（推荐，与函数内一致 + 与 Swift 一致）
2. **或完全不做检测**（与 Java 一致，最简单）
3. **spec 应明确**：哪些场景属于"编译期可检测"

### 问题 V：整数 vs 浮点除零行为完全不同 ⭐ LOW ⭐ NEW

**用例：** 022~030（整数）+ 浮点用例 TYP_03_06_03_021~030

**实测对比：**

| 操作 | 整数 | 浮点 |
|------|------|------|
| 字面量 `T / 0` | ❌ 编译错误 | ✅ 通过 → Infinity |
| `% 0` | ❌ 编译错误 | ✅ 通过 → NaN |
| const 检测 | 函数内编译错误 / 模块级通过 | 全部通过 |
| 运行时除零 | 抛 ArithmeticError | 返回 Inf/NaN |

**评价：**
- 整数 vs 浮点完全不同行为有其合理性（浮点遵循 IEEE 754）
- 但用户切换数值类型时容易意外
- spec 描述清晰（§15.17.2 分别说明两者）
