## 一、与业界静态语言的差异点

**未发现设计问题。所有行为均符合 ArkTS 规范第 8.1 节。**

第 8.1 节是一个概念性/定义性章节，建立了正常语句完成（normal completion）与突然语句完成（abrupt completion）的基础概念。这些概念与标准编程语言语义一致，且在所有 9 个测试用例中均被正确实现。

### 逐项验证结果

| 概念 | ArkTS 行为 | 设计问题 | 严重性 |
|------|-----------|---------|--------|
| 正常完成 | 语句按预期执行所需操作，无错误发生 | 无 | -- |
| 突然完成 | 语句执行导致错误被抛出 | 无 | -- |
| throw 语义 | 仅 Error 及其子类可被抛出 | 无 | -- |
| break 语义 | 必须出现在循环或 switch 内部 | 无 | -- |
| continue 语义 | 必须出现在循环内部 | 无 | -- |
| try-catch 处理 | catch 捕获突然完成，正常流程恢复 | 无 | -- |
| finally 执行 | finally 始终执行，即使发生突然完成 | 无 | -- |

## 四、跨语言对比结论

基于 ArkTS vs Java (JLS SE21 §14) vs Swift (5.x) 的详细对比：

### 关键差异矩阵

| 方面 | ArkTS | Java (SE21) | Swift (5.x) |
|--------|-------|-------------|-------------|
| **可抛出类型** | 仅 `Error` 及其子类 | 任意 `Throwable` 子类 | 任意遵循 `Error` 协议的类型 |
| **受检异常** | 不适用（无受检异常） | 受检异常必须声明或捕获 | 无受检异常 |
| **抛出非错误字面量** | 编译时错误 | 编译时错误（必须是 `Throwable`） | 编译时错误（必须遵循 `Error`） |
| **try-catch 作为表达式** | 仅语句 | 仅语句 | `do-catch` 可用作表达式 |
| **finally 等效机制** | `finally` 块 | `finally` 块 | `defer`（基于作用域，不绑定于 try） |
| **带标签的 break/continue** | 支持标签 | 支持标签 | 不支持 |
| **switch 中的 break** | 贯穿 + break | 贯穿 + break | 无隐式贯穿；不需要 break |
| **不可达代码检测** | 编译期检测 | 编译期检测 | 编译期检测 |

### 综合评分

| 维度 | ArkTS vs Java | ArkTS vs Swift |
|------|:------------:|:--------------:|
| 概念对齐度 | 10/10 | 10/10 |
| 语法相似度 | 9/10 | 8/10 |
| 错误类型系统 | 8/10 | 9/10 |
| 运行时行为 | 10/10 | 10/10 |
| 编译时检查 | 10/10 | 10/10 |
| **平均** | **9.4/10** | **9.4/10** |

### 核心结论

1. **强对齐**：ArkTS 第 8.1 节在概念上与 Java (JLS SE21 第 14 章) 和 Swift (5.x) 高度对齐。三种语言均将正常完成定义为默认执行路径，将突然完成定义为由 throw、break、continue 或 return 触发的异常路径。

2. **ArkTS 更接近 Java**：在语法和结构上，ArkTS 与 Java 非常接近——共享 `try-catch-finally` 模式、带标签的 break/continue 以及通用的块结构语句模型。Swift 则以 `do-catch` 和 `defer` 产生偏离。

3. **finally 与 defer 的差异**：Swift 没有 `finally` 块，而是使用 `defer`（在作用域退出时执行）。这是一种更通用的机制，但为突然完成后的清理提供了相同的保证。ArkTS 和 Java 的 `finally` 在结构上绑定于 `try`，语义等价。

4. **无语义缺口**：所有为 ArkTS 第 8.1 节编写的测试用例在 Java 和 Swift 中均有直接的语义等价物。未发现设计层面的差异。

5. **ArkTS 的 throw 限制具有可比性**：ArkTS 将 throw 限制为 `Error` 及其子类，这与 Java 要求 throw 必须接受 `Throwable` 子类的限制以及 Swift 要求抛出类型必须遵循 `Error` 协议的限制具有可比性。

本节未发现设计问题，以下建议聚焦于测试覆盖增强和工具链完善：

### 中期（后续迭代）

   - finally 块自身抛出错误时的行为

### 长期（集成与深度测试）

5. **集成测试**：增加更复杂的嵌套场景，验证突然完成跨多个函数调用层的传播行为。
