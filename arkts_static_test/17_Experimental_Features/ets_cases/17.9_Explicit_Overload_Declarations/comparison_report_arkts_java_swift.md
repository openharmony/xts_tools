# 17.9 Explicit Overload Declarations - ArkTS vs Java vs Swift 对比报告

**生成日期：** 2026-06-23
**规范来源：** ArkTS Static Spec §17.9, Java JLS SE21 §8.4.9/§15.12, Swift Language Reference (Functions/Methods)
**测试基础：** 73 个用例（38 compile-pass + 24 compile-fail + 11 runtime 真实执行）

---

## 一、概览：三语言定位

| 语言 | 重载机制定位 | 设计哲学 |
|------|------------|---------|
| **ArkTS** | 显式 overload 关键字声明有序重载集 | 编译时多态，显式声明重载关系 |
| **Java** | 隐式重载（同名方法不同签名即可） | 传统 OOP 隐式重载，无需显式声明 |
| **Swift** | 隐式重载 + 外部参数名区分 | 现代化重载，参数标签参与签名 |

---

## 二、核心对比矩阵

### 2.1 函数重载

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 声明方式 | `overload name { f1, f2 }` | 自然同名即可 | 自然同名即可 |
| 重载集可见性 | 显式声明，一目了然 | 隐式，需查找所有同名函数 | 隐式，需查找所有同名函数 |
| 一个实体多个重载集 | ✅ 支持 | N/A (隐式) | N/A (隐式) |
| 导出约束 | 所有重载函数必须导出 | 无约束 | 无约束 |
| 重载名与函数同名 | ✅ 支持（函数须在列表中） | 直接同名 | 直接同名 |
| 空重载列表 | 编译器允许（⚠️ spec 说禁止） | N/A | N/A |

### 2.2 方法重载

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 声明方式 | `overload name { m1, m2 }` | 同名方法不同签名 | 同名方法不同签名 |
| static/instance 隔离 | ✅ 编译时检查 | 自动区分 | 自动区分 |
| async 隔离 | ✅ 编译时检查 | ❌ 无 async 概念 | ✅ async 自动区分 |
| 访问修饰符约束 | ✅ public 不能含 private | ❌ 无约束 | Swift 有访问控制 |
| 重写规则 | 必须列出所有父类方法 | 自然覆盖 | 自然覆盖 + override 关键字 |
| 特殊方法 ($_get 等) | ✅ 支持 | N/A | N/A |

### 2.3 接口重载

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 接口中声明 | `overload name { m1, m2 }` | 接口方法同即可 | 协议方法同名即可 |
| 实现类必须覆盖 | 可选（继承或覆盖） | 必须实现 | 必须实现 |
| 多接口同名冲突 | ✅ 编译时检测并报错 | ❌ 需手动处理 | ❌ 需手动处理 |
| 实现类添加重载 | ✅ 可添加额外方法 | ✅ 自然支持 | ✅ 自然支持 |

### 2.4 构造函数重载

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 声明方式 | `overload constructor { c1, c2 }` | 多个 constructor 即可 | 多个 init 即可 |
| 匿名构造函数 | 隐式在列表首位 | 普通构造函数 | 普通 init |
| 命名构造函数 | 需列在 overload 中 | N/A（无此概念） | N/A（无此概念） |
| 每类限制 | spec: 仅一个 overload 声明（⚠️ 编译器未强制） | 无限制 | 无限制 |
| 空列表 | 编译器允许（⚠️ spec 说禁止） | N/A | N/A |

## 三、关键发现

### 发现 1：ArkTS 的唯一显式重载机制

ArkTS 是唯一需要显式声明重载集的语言。Java 和 Swift 都是隐式重载——只要同名方法有不同的参数签名就自动构成重载关系。

**优势：**
- 重载关系一目了然，无需搜索代码库
- 编译时可检测重载集完整性

**劣势：**
- 需要额外的声明代码
- 添加新重载时必须同时更新 overload 声明

### 发现 2：ArkTS 的重载安全约束良好

| 约束 | ArkTS | Java | Swift |
|------|-------|------|-------|
| static/instance 隔离 | ✅ | ❌ | ❌ (无 static 概念) |
| async 隔离 | ✅ | N/A | ✅ |
| 访问修饰符约束 | ✅ | ❌ | ✅ (访问控制) |
| 导出一致性检查 | ✅ | ❌ | ❌ |
| 重写完整性检查 | ✅ | ❌ (隐式) | ✅ (override 关键字) |
| 多接口冲突检测 | ✅ | ❌ | ❌ |

### 发现 3：构造函数重载语法差异显著

ArkTS 的构造函数重载语法与 Java/Swift 完全不同：

```typescript
// ArkTS - 命名构造函数 + 显式 overload
class Foo {
  constructor fromInt(a: int) { ... }
  constructor fromStr(a: string) { ... }
  overload constructor { fromInt, fromStr }
  // 匿名构造函数隐式在首位
}
```

```java
// Java - 自然重载
class Foo {
    Foo(int a) { ... }
    Foo(String a) { ... }
}
```

```swift
// Swift - init 重载
class Foo {
    init(_ a: Int) { ... }
    init(_ a: String) { ... }
}
```

### 发现 4：SPEC 不一致（3 处）

| 编号 | 描述 | 严重度 |
|------|------|--------|
| S1 | 空 overload 列表应当报错，实际编译通过 | MEDIUM |
| S2 | 空 constructor overload 列表应当报错，实际编译通过 | MEDIUM |
| S3 | 每类只能一个 constructor overload 声明，实际允许多个 | LOW |

## 六、核心结论

1. **ArkTS 是唯一显式声明重载集的语言**，提供了良好的编译时安全性
2. **Java 最简洁直观**但缺乏重载安全约束
3. **Swift 中间态**，参数标签 + override 关键字提供部分安全性
4. **ArkTS 构造函数重载设计特殊**，命名构造函数 + 显式列表 + 匿名隐式优先
5. **3 处 SPEC 不一致**需要在实现或规范中协调
